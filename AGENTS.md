# ZeroDivida Agent Handbook

Living guide for agents working on the Base44 ZeroDivida workspace. Keep builds deterministic, preserve the design language, and document any deviation from these rules inside this file before shipping.

## Quick Context
- Stack: React 18 + Vite 6 + Tailwind + shadcn/ui with React Query and Firebase-backed Base44 SDK mocks.
- Target runtime: Node 18.18+ and npm (package-lock.json is authoritative; do not mix package managers).
- Pages are auto-wired through `src/pages.config.js`; only `mainPage` is editable.
- Authentication and app telemetry flow through `src/lib/AuthContext.jsx`, `NavigationTracker.jsx`, and the `base44` client defined in `src/api/base44Client.js`.
- Visuals lean on a dark palette (`#0B0F19` background, emerald highlights) and Portuguese copy; keep additions consistent unless requirements say otherwise.

## Setup Checklist
1. Run `npm install` after cloning or switching branches; avoid `npm update` unless dependency work is intentional.
2. Copy `.env.example` to `.env` (or `.env.local`) and fill every `VITE_FIREBASE_*` variable plus `VITE_BASE44_*` values referenced in `src/lib/app-params.js`.
3. Use `npm run dev` for local work; Vite serves on http://localhost:5173 by default with hot module reload.
4. If Firebase credentials are absent you still get mocked data, but realtime sync will log warnings; hook up real credentials for end-to-end testing.
5. Keep `node_modules` intact—lint and typecheck scripts rely on ESLint 9 flat config and TypeScript 5.8 even though most files are JS.

## Project Topology
- `src/pages/*.jsx` define routed views (Dashboard, Debts, Incomes, Reports). Use PascalCase filenames so they auto-register.
- `src/components/` holds feature blocks (dashboard, debts, income, reports) plus reusable primitives under `ui/` (shadcn-derived, lint-ignored).
- `src/lib/` contains cross-cutting utilities (auth, query client, Firebase config, nav tracking, app params).
- `src/hooks/` exposes `useFirebaseRealtime`, `useMobile`, etc.; keep hooks pure and side-effect boundaries inside `useEffect`.
- `entities/` documents Firestore collections; mirrored in `src/mocks/firebase-sdk.js` within `createClient`.

## Routing & Navigation Rules
- `pages.config.js` is regenerated by Base44. Never add imports manually; only switch the `mainPage` string when changing the landing screen.
- React Router lives inside `App.jsx`. Stick to declarative `<Routes>` within `AuthenticatedApp` and wrap content in `LayoutWrapper` so the shared chrome stays consistent.
- Use `createPageUrl` from `src/utils/index.ts` when linking via `Link` to guarantee slug formatting matches the nav menu.
- `NavigationTracker` logs page transitions through `base44.appLogs`. Keep those calls best-effort and never block rendering if they fail.

## State, Data, and Realtime
- Prefer React Query for async state; use array query keys (`["debts"]`) and invalidate using `queryClient.invalidateQueries({ queryKey: [...] })` to benefit from partial matching.
- For Firestore-backed lists, call `useFirebaseRealtime(entityName, queryKey, orderByField)` to subscribe and hydrate caches instead of wiring your own `onSnapshot`.
- Mutations should optimistically close dialogs or reset UI state only after `onSuccess` to keep UX predictable (see Dashboard create/update/delete flows).
- Keep derived calculations (progress, currency formatting) inside memoized helpers or inline functions near their usage; avoid storing them in React state.

## API and Entity Access
- The `base44` client is instantiated via `createClient` (Firebase mock). Entities are accessed through `base44.entities.{Debt,Income,Payment}` with CRUD helpers.
- Add a new entity by defining the collection in Firebase and registering it inside `createClient`; keep the method surface identical (list/subscribe/create/update/delete/get).
- For authenticated routes, use `useAuth()` from `AuthContext`. It returns `user`, `isAuthenticated`, `authError`, and helpers like `logout` and `navigateToLogin`.
- REST-like calls use the Axios shim from `src/mocks/firebase-sdk.js`. Keep those asynchronous wrappers resilient—catch and log instead of throwing uncaught errors to React tree.

## Styling and Theming
- Tailwind is the primary styling tool. Follow the existing convention of chaining utility classes with gradients, opacity, and `group-hover` states to preserve the neon-dark aesthetic.
- Theme tokens come from CSS variables defined in `tailwind.config.js`. Extend tokens before sprinkling arbitrary hex values to keep colors centralized.
- Shared primitives live under `src/components/ui`. Favor those components (`<Button>`, `<Badge>`, `<Dialog>`, `<Toaster>`) over raw HTML to inherit accessibility and theme props.
- Animations rely on Framer Motion; keep durations under 400ms, use `initial/animate` pairs, and stagger sequences with small delays (0.03–0.08) to match existing cards.

## Copy, Localization, and Formatting
- User-facing text is currently in Portuguese; maintain the language unless specs require localization, and keep diacritics accurate.
- Currency formatting goes through `Intl.NumberFormat('pt-BR', { style: 'currency', currency: 'BRL' })`. Reuse helpers like `formatCurrency` in `DebtCard.jsx` instead of duplicating logic.
- Dates default to ISO strings stored in Firestore; format for display with `date-fns` or `moment` as needed, but keep persisted values ISO to stay compatible with Firebase queries.

## Imports and Module Conventions
- Order imports as: external packages, alias-based (`@/`), then relative paths. Group icon imports (lucide-react) together and collapse multi-icons into a single destructuring line.
- Use the `@/*` alias defined in `vite.config.js` and `jsconfig.json` to avoid brittle relative paths; update `jsconfig` whenever new top-level directories need inclusion.
- Avoid default-exporting data utilities; named exports (`export function useFirebaseRealtime`) improve tree-shaking, while React components remain default exports for readability.
- Remove unused imports aggressively—`eslint-plugin-unused-imports` raises hard errors for leftovers, so run `npm run lint` before committing.

## Types, Formatting, and Semantics
- Even though most files are `.jsx`, keep TypeScript-friendly patterns: annotate helper signatures inline (`function createPageUrl(pageName: string)`), prefer explicit returns, and respect `eslint` parser settings.
- Formatting: two-space indentation, trailing commas in objects/arrays where valid, semicolons at the end of statements, double quotes for strings, and template literals for class concatenation.
- Use descriptive const names in SCREAMING_SNAKE_CASE for config maps (`CATEGORY_CONFIG`, `STATUS_CONFIG`) and camelCase for functions/props/state.

## Component Structure and Naming
- Write components as `function ComponentName(props) {}` with an exported default at the bottom. Keep files focused: one component per file unless rendering tiny stateless helpers.
- Derive UI state via `useState` near the top of the component and group related `useMutation` or `useQuery` declarations together for readability.
- Keep modals/forms co-located with the page that triggers them (`DebtForm` inside debts folder). When a component graduates to shared usage, migrate it into `components/ui` and document props in the file header.

## Error Handling and Logging
- Network and auth errors flow through `AuthContext`; never throw inside render bodies. Instead, set contextual `authError` and branch UI as shown in `App.jsx`.
- Wrap `base44` CRUD calls in `try/catch` within async handlers or mutation hooks. Report failures via `console.error` plus user-facing toasts (`@/components/ui/toaster` or `@/components/ui/sonner`).
- Keep logging side effects (NavigationTracker, realtime subscriptions) best-effort—swallow errors and leave a comment if ignoring is intentional.

## Forms and Validation
- Use `react-hook-form` with `zod` or inline resolvers defined in `@hookform/resolvers` when building complex forms like debts/payments; keep schema definitions near the component for discoverability.
- Inputs, selects, date pickers, and sliders should use the shadcn components under `components/ui` to ensure consistent focus rings and error states.
- Surface validation errors inline under fields and optionally mirror them in toast notifications for destructive actions.

## Accessibility and UX Notes
- Maintain keyboard support: buttons and links should always be reachable, modals must trap focus (shadcn/dialog already enforces this if configured correctly).
- Provide aria labels for icon-only buttons (menu toggles, close icons) and ensure text contrast meets WCAG AA on the dark background.
- Use motion sparingly on reduced-motion devices; wrap major Framer Motion animations with `prefers-reduced-motion` checks if you add new ones.

## Performance Guardrails
- Memoize expensive derived data (aggregations for charts) or compute them inside `useMemo` when fed to large lists.
- Paginate or virtualize when rendering more than ~25 cards; `react-resizable-panels` and `react-virtualized` can help if needed.
- Avoid unnecessary re-renders by keeping props stable; pass IDs or minimal slices of state into deeply nested components.

## Environment and Secrets
- `.env` should never be committed. Use `.env.example` to document new variables and keep keys prefixed with `VITE_` so Vite exposes them safely.
- `src/lib/app-params.js` reads URL params (e.g., `access_token`) and persists them to `localStorage`. When debugging, append `?clear_access_token=true` to reset state.
- Always confirm Firebase credentials correspond to the right project before running destructive scripts.

## Git and Review Habits
- Branch from the latest main, keep commits focused, and never rewrite history unless the maintainer requests it.
- Run `npm run lint` and `npm run typecheck` before opening a PR; attach screenshots for UI-facing changes.
- Document env or schema updates inside the PR description so Base44 Builder stays in sync.

## Tooling Tips
- Enable React Query Devtools while developing to inspect cache state, invalidate keys, and watch realtime sync updates from `useFirebaseRealtime`.
- Use the browser Performance panel plus React Profiler for long lists; capture flame charts before optimizing animations.
- Tailwind IntelliSense (VS Code) speeds up class discovery—keep `tailwind.config.js` open if you need to extend tokens.
- Rely on ESLint's flat config for quick feedback; if rules feel stale, update this handbook after adjusting the config.

## Analytics and Telemetry
- `NavigationTracker` is the single place we ping `base44.appLogs.logUserInApp`; new tracking events should reuse the same client to keep auth context centralized.
- Keep telemetry fire-and-forget—wrap in `try/catch` and swallow network failures so UI rendering is never blocked.
- When adding screen names, sync them with Base44 analytics dashboards and update this doc so future agents can trace naming.

## Troubleshooting Cheatsheet
- **Realtime not updating**: confirm Firebase credentials, ensure entity names in `createClient` match Firestore collections, and restart dev server after `createClient` edits.
- **Auth stuck on spinner**: inspect `localStorage` for `base44_access_token`, then use `?clear_access_token=true` to reset before reloading.
- **Lint errors inside shadcn/ui**: those files are ignored; if you move code out of `components/ui`, make sure it passes `npm run lint`.
- **Imports unresolved**: run `npm install` to rehydrate `node_modules` and verify `@` alias still points to `./src` in `vite.config.js`.

## Adding or Updating Dependencies
- Stick with `npm install <pkg>` so `package-lock.json` stays authoritative and deterministic.
- Check bundle impact for heavy libraries (charts, editors) by running `npm run build` and inspecting the output size in `dist/assets`.
- Document why the dependency is needed inside PR notes and update this handbook if it changes our workflow.

## Manual Test Checklist (Current Substitute for Automated Tests)
1. `npm run dev`, then load `/` and confirm Dashboard data renders (cards, charts, realtime updates).
2. Navigate to `/Debts`, create a debt, edit it, add a payment, and ensure the status/progress updates.
3. Visit `/Incomes` and `/Reports` to confirm routing and layout chrome remain intact.
4. Toggle the mobile menu and ensure nav links close the drawer.
5. Manually clear `localStorage` and reload to verify auth fallback UI (loading spinner, user not registered toast) still behaves.
6. Run `npm run build` and `npm run preview` to ensure the production bundle starts without console errors.
